# 让项目脱颖而出 — 想法记录

这是 **agentic** 题：评分看 agent 怎么推理、给了什么工具、行为是否被调教得稳定可靠。
脚手架现状的结构性短板（也是发力点）：
- agent 看不到设计：只有 `WebFetch`（文字/markdown）+ `write_html`/`read_html`，从没"看见"目标站或自己的产物。
- `write_html` 整文档覆写：多轮编辑慢、易回退、易丢状态。
- 没有视觉闭环：改完无法自评"像不像"。
- 没有设计 token / 素材抽取：颜色、字体、间距全靠猜。

## 想法（按性价比排序）

1. **给 agent「眼睛」——视觉闭环（最高杠杆）**
   - Playwright 对目标 URL 截全屏图 + 抓 DOM/computed styles，作为多模态输入喂模型。
   - 对自己生成的 `output/index.html` 也截图，让 agent 看结果 → 与原图对比 → 自我修正。
   - 把"凭文字脑补"变成"看图复刻 + 视觉 diff 迭代"，保真度质变，也是 demo 最亮的一幕。

2. **抽取「设计 token」而非死复刻**
   - 工具：从目标站提取调色板、字体栈、字号梯度、圆角、阴影、间距节奏，落成 CSS 变量（`:root`）。
   - 既"几乎一模一样"又天然可改；"换成我们品牌色"=改几个变量。回应 README 的 clean / customizable。

3. **细粒度编辑工具，避免整文档覆写**
   - 加 `edit_section` / `replace_block`（按区块或选择器改）或 `patch_html`。
   - 改哪块只动哪块，多轮更快、更稳、可回退。

4. **资产处理**
   - 抓 hero 图 / logo / 图标并本地化（或换占位图），避免外链失效；让"换 hero 图"真正成立。

5. **可靠性 / agent 行为调教（直接对应评分点）**
   - 系统提示固化工作流：抓取理解 → 抽 token → 分区块搭骨架 → 截图自检 → 修正。
   - 结构化进度提示（抓取/分析配色/还原 hero/自检差异），让推理过程可见。
